# Guía de documentación de Skills

## Propósito

Esta guía se usa cuando el usuario quiere documentar, formalizar, estructurar o completar la información de un Skill.

La documentación formal de un Skill debe usar como fuente obligatoria el archivo:

```text
template-documentar_skill.md
```

---

## Cuándo usar esta guía

Usa esta guía cuando el usuario diga cosas como:

- “Documenta este Skill.”
- “Crea la documentación de este Skill.”
- “Estructura este Skill.”
- “Formaliza este Skill.”
- “Completa la ficha de este Skill.”
- “Organiza esta idea como documentación de Skill.”
- “Usa el template para documentar este Skill.”

---

## Regla principal

No inventes una estructura alternativa para documentar un Skill.

Consulta y usa la estructura vigente de:

```text
template-documentar_skill.md
```

---

## Proceso para documentar un Skill

Cuando se active una solicitud de documentación:

1. Consulta `template-documentar_skill.md`.
2. Usa su estructura vigente como formato base.
3. Completa cada sección con la información disponible.
4. No elimines secciones importantes.
5. No inventes datos críticos.
6. Si falta información, marca el campo como pendiente.
7. Si propones contenido, indica que es una propuesta inicial.
8. Mantén el resultado en Markdown.
9. Entrega la documentación lista para copiar, pegar y guardar.

Antes de completar el template, identifica si el usuario entregó:

- Una idea inicial.
- Un diseño parcialmente definido.
- Un Skill ya existente.
- Una documentación que necesita revisión.

El nivel de certeza de la documentación debe corresponder con el nivel de información disponible.

---

## Manejo de información faltante

Si falta información crítica, puedes:

- Formular preguntas específicas.
- Marcar una sección como pendiente.
- Declarar supuestos explícitos.
- Crear una versión preliminar y advertir qué falta validar.

Usa estas marcas:

- `Pendiente`: cuando no hay información suficiente.
- `Supuesto`: cuando se propone una interpretación razonable.
- `Por validar`: cuando la información parece probable, pero requiere confirmación del usuario.

No completes campos críticos con frases genéricas. Es mejor dejar un pendiente claro que producir documentación aparentemente completa pero débil.

---

## Criterios de completitud documental

Una documentación de Skill se considera sólida cuando:

- El nombre permite entender la capacidad principal.
- El propósito explica qué problema resuelve.
- El usuario objetivo no es genérico.
- El contexto de uso indica cuándo se activa.
- El objetivo operativo está redactado como acción concreta.
- Las entradas esperadas son claras y separan obligatorias de opcionales cuando aplique.
- Las salidas esperadas indican formato, contenido y nivel de detalle.
- El flujo de trabajo tiene pasos secuenciales.
- Las reglas de comportamiento incluyen límites.
- Los criterios de calidad permiten evaluar la salida.
- Los casos de uso muestran entradas y resultados esperados.
- Los recursos necesarios están listados o se marca que no requiere recursos adicionales.
- Los riesgos o ambigüedades no se ocultan.
- El checklist final permite validar si el Skill está listo.

---

## Resultado esperado

La respuesta debe entregar un documento formal de Skill en Markdown, basado en `template-documentar_skill.md`.

Debe ser claro, completo, reutilizable y listo para guardar como documentación del Skill.

Si la documentación es preliminar, el título o una nota inicial debe indicarlo claramente.
