# Guía de evaluación de Skills de ChatGPT

## Propósito

Esta guía se usa cuando el usuario quiere evaluar si una tarea, idea, proceso o documentación debería convertirse en Skill o si un Skill existente está bien diseñado.

---

## Criterios para identificar si algo debería convertirse en Skill

Cuando el usuario describa una tarea, analiza si conviene convertirla en Skill usando estos criterios.

La evaluación debe distinguir entre tres preguntas distintas:

1. ¿La tarea tiene sentido como Skill?
2. ¿La idea está suficientemente definida?
3. ¿La documentación o diseño actual está listo para usarse?

No confundas una buena idea con un Skill listo. Una idea puede ser prometedora y aun así necesitar mejor definición.

---

## Alta probabilidad de convertirse en Skill

La tarea tiene alta probabilidad de convertirse en Skill si:

- Se repite con frecuencia.
- Tiene pasos claros.
- Requiere consistencia.
- Usa una plantilla o formato fijo.
- Necesita seguir reglas específicas.
- Se beneficia de ejemplos o archivos de referencia.
- Puede automatizar parte del trabajo.
- Tiene entradas y salidas relativamente estables.
- Requiere validaciones o criterios de calidad recurrentes.

---

## Baja probabilidad de convertirse en Skill

La tarea tiene baja probabilidad de convertirse en Skill si:

- Es única o muy puntual.
- Cambia demasiado cada vez.
- No tiene un flujo claro.
- Depende principalmente de criterio creativo abierto.
- No requiere consistencia.
- Puede resolverse con una simple instrucción de conversación.
- No necesita recursos, reglas ni referencias reutilizables.

---

## Matriz de evaluación de oportunidades para Skills

Cuando el usuario quiera evaluar ideas de Skills, usa esta matriz:

| Criterio | Pregunta guía | Puntaje 1-5 | Qué significa un puntaje alto |
|---|---|---|---|
| Frecuencia | ¿La tarea se repite muchas veces? | | Se usa de forma recurrente |
| Claridad | ¿El proceso tiene pasos claros? | | Puede convertirse en flujo |
| Valor | ¿Ahorra tiempo o mejora calidad? | | Tiene impacto práctico visible |
| Consistencia | ¿Necesita resultados estandarizados? | | Requiere formato o criterios estables |
| Reutilización | ¿Sirve para más de un caso? | | Aplica a escenarios similares |
| Complejidad | ¿Tiene suficiente complejidad para justificar un Skill? | | Supera lo que resolvería un prompt simple |
| Recursos | ¿Necesita plantillas, archivos, referencias o scripts? | | Se beneficia de materiales reutilizables |

Interpretación:

| Puntaje total | Interpretación |
|---|---|
| 25-35 puntos | Muy buen candidato para Skill. |
| 15-24 puntos | Puede funcionar como Skill si se define mejor. |
| Menos de 15 puntos | Probablemente basta con una instrucción, prompt o plantilla. |

---

## Evaluación de un Skill existente

Cuando el usuario pida revisar o auditar un Skill, evalúa:

| Dimensión | Pregunta guía |
|---|---|
| Claridad del objetivo | ¿El Skill tiene un propósito claro? |
| Usuario objetivo | ¿Está claro quién lo usará? |
| Entradas | ¿Se sabe qué información necesita? |
| Salidas | ¿Se sabe qué debe entregar? |
| Flujo | ¿Tiene pasos lógicos y ejecutables? |
| Reglas | ¿Define qué debe hacer y qué debe evitar? |
| Calidad | ¿Incluye criterios para validar el resultado? |
| Reutilización | ¿Puede aplicarse en varios casos similares? |
| Ambigüedad | ¿Hay instrucciones confusas o incompletas? |
| Escalabilidad | ¿Puede crecer sin volverse inmanejable? |

Añade una clasificación de madurez:

| Nivel | Estado | Señal |
|---|---|---|
| Idea inicial | Todavía conceptual | Hay propósito, pero faltan entradas, salidas o flujo |
| Diseño preliminar | Parcialmente estructurado | Hay objetivo y flujo, pero faltan reglas o criterios |
| Documentación funcional | Listo para prueba | El template está completo con pocos pendientes |
| Skill robusto | Listo para uso recurrente | Tiene reglas, ejemplos, límites y criterios verificables |

---

## Resultado esperado de una evaluación

La respuesta debe incluir:

1. Diagnóstico general.
2. Fortalezas.
3. Debilidades.
4. Riesgos.
5. Puntaje o valoración.
6. Recomendación final.
7. Siguientes pasos.

Cuando uses puntajes, explica brevemente el motivo de cada criterio. No entregues solo el total.

Si la evaluación concluye que no conviene un Skill, recomienda la alternativa más adecuada: prompt, plantilla, proyecto, GPT personalizado o documentación interna.

---

## Recomendaciones finales posibles

Usa una de estas conclusiones:

- Muy buen candidato para Skill.
- Puede funcionar como Skill si se define mejor.
- Conviene primero documentar mejor el proceso.
- Conviene usar un prompt o plantilla.
- Conviene usar un proyecto.
- Conviene usar un GPT personalizado.
- No conviene convertirlo en Skill por ahora.

---

## Señales de alerta

Marca como riesgo cuando:

- El Skill intenta cubrir demasiadas tareas distintas.
- El usuario objetivo no está definido.
- Las entradas dependen de información que normalmente no estará disponible.
- La salida esperada no tiene formato claro.
- El flujo requiere decisiones humanas no descritas.
- Las reglas se contradicen entre sí.
- La documentación mezcla explicación conceptual con instrucciones operativas.
- El Skill depende de capacidades técnicas no confirmadas.
