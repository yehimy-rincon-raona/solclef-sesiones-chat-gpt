# Template Genérico de Análisis de Discovery para Skill

## Propósito del documento

Este template sirve como insumo genérico para que un Skill analice entrevistas, notas, formularios o transcripciones de discovery y produzca una salida estructurada, accionable y priorizada.

Está diseñado para usarse como documento adjunto de referencia en ejercicios de diseño, documentación o prueba de Skills. No contiene datos privados ni referencias internas de una empresa específica.

---

## Uso recomendado

- Adjuntar este documento como referencia cuando se diseñe o pruebe el Skill.
- Usarlo como formato esperado de salida, no como formulario privado de una empresa.
- Adaptar los campos según el contexto del proyecto sin cambiar la lógica general.
- Pedir al Skill que complete solo los campos que pueda sustentar con evidencia disponible.
- Marcar como `Pendiente`, `Supuesto` o `Por validar` cualquier información insuficiente.

---

## Principios de uso del template

- Separar información explícita de inferencias.
- Diferenciar dolores, hallazgos, oportunidades y recomendaciones.
- Priorizar oportunidades con criterios consistentes.
- Evitar recomendaciones no sustentadas en la evidencia del discovery.
- Mantener una salida útil para stakeholders, equipos de producto, consultoría o preventa.

---

# Estructura de salida recomendada para el Skill

1. Resumen ejecutivo del discovery.
2. Información normalizada del levantamiento.
3. Hallazgos clave.
4. Dolores y necesidades identificadas.
5. Oportunidades detectadas.
6. Matriz de priorización.
7. Riesgos, vacíos y preguntas pendientes.
8. Recomendación de próximos pasos.

| Impacto | Valor potencial para negocio, operación, experiencia, productividad o indicadores. | Alto / Medio / Bajo |
| Evidencia | Qué tan sustentada está la oportunidad en la entrevista o insumo recibido. | Alta / Media / Baja |
| Urgencia | Presión temporal, criticidad o necesidad de resolver pronto. | Alta / Media / Baja |
| Viabilidad funcional | Claridad sobre lo que debe hacer la solución y cómo la usarían los usuarios. | Alta / Media / Baja |
| Viabilidad técnica | Factibilidad según datos, sistemas, integraciones, permisos o recursos disponibles. | Alta / Media / Baja |
| Esfuerzo | Complejidad aproximada de diseño, implementación, integración o validación. | Alto / Medio / Bajo |
| Riesgo | Incertidumbres que podrían bloquear, encarecer o retrasar la solución. | Alto / Medio / Bajo |
| Alineación estratégica | Conexión de la oportunidad con el objetivo del proyecto o necesidad del negocio. | Alta / Media / Baja |

# 15. Reglas de priorización para el Skill

1. Una oportunidad solo puede tener prioridad alta si tiene impacto alto o medio, evidencia alta o media, y viabilidad funcional o técnica al menos media.
2. Si la evidencia es baja, la oportunidad debe marcarse como `Por validar`, aunque parezca valiosa.
3. Si el riesgo es alto, la oportunidad no debe recomendarse como prioritaria sin incluir una acción de mitigación.
4. Si el esfuerzo es alto y el impacto es bajo, la oportunidad debe quedar en prioridad baja.
5. Si la oportunidad depende de datos, sistemas, licencias, permisos o decisiones no confirmadas, debe marcarse como `Por validar`.
6. El Skill debe diferenciar entre información explícita, inferencia razonable, supuesto y recomendación.
7. El Skill no debe proponer IA si el problema puede resolverse mejor con automatización simple, integración, rediseño de proceso, reportería o mejora UX/UI.
